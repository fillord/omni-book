# Plan 12-01: Schema migration + Kaspi file deletion + vercel.json cleanup

## Goal
Remove all Kaspi-specific infrastructure: delete dead files, run Prisma migration to drop Kaspi fields from schema, remove the pending-payments cron from vercel.json.

## Context
Phase 9 introduced Kaspi Pay as a mock. Phase 10 introduced a mock platform payment QR. Neither was ever connected to a real Kaspi API. We are now pivoting to Paylink.kz (Phase 12-03) and WhatsApp (Phase 12-02). This plan handles the lowest-risk first: pure deletions and schema changes that have no UI impact yet.

## Requirements covered
- PIV-01: Delete Kaspi files + remove cron
- PIV-02: Schema cleanup

---

## Tasks

### Task 1 — Delete Kaspi-specific source files
Delete the following files:
- `lib/payments/kaspi.ts`
- `app/api/webhooks/kaspi/route.ts`
- `app/api/cron/pending-payments/route.ts`

These are standalone files with no imports from other app code that would break. Verify by checking what imports them before deleting.

**Verification:**
```bash
grep -r "from.*payments/kaspi\|webhooks/kaspi\|cron/pending-payments" --include="*.ts" --include="*.tsx" . | grep -v node_modules | grep -v .planning | grep -v ".claude/worktrees"
```
Expected: `app/api/bookings/route.ts` imports `kaspi.ts` → handled in Plan 12-02. The webhook and cron files are not imported anywhere else.

Delete in order:
1. `lib/payments/kaspi.ts`
2. `app/api/webhooks/kaspi/route.ts`
3. `app/api/cron/pending-payments/route.ts` (entire directory if only contains `route.ts`)

### Task 2 — Remove pending-payments cron from vercel.json
Read `vercel.json`, find the `/api/cron/pending-payments` entry, and remove it.

```json
// Remove this entry from crons array:
{ "path": "/api/cron/pending-payments", "schedule": "*/5 * * * *" }
```

### Task 3 — Prisma schema migration: remove Kaspi fields from Tenant
In `prisma/schema.prisma`, remove from model `Tenant`:
```prisma
requireDeposit   Boolean  @default(false)
depositAmount    Int      @default(0)
kaspiMerchantId  String?
kaspiApiKey      String?
```

**Note:** `requireDeposit` and `depositAmount` also exist on the `Service` model — do NOT remove them from Service. Only remove from Tenant.

### Task 4 — Prisma schema migration: remove Kaspi fields from Booking
In `prisma/schema.prisma`, remove from model `Booking`:
```prisma
paymentInvoiceId  String?
paymentExpiresAt  DateTime?
```

Also remove the composite index that references `paymentExpiresAt`:
```prisma
@@index([status, paymentExpiresAt])
```

### Task 5 — Prisma schema migration: clean up PlatformPayment model
In `prisma/schema.prisma`, update the `PlatformPayment` model to replace mock fields with Paylink fields. Change:
```prisma
  mockQrCode  String?
  mockPaylink String?
```
to:
```prisma
  paylinkOrderId String?
  paylinkUrl     String?
```

This prepares the model for Plan 12-03 where the real Paylink adapter is wired.

### Task 6 — Generate and run Prisma migration
```bash
npx prisma migrate dev --name remove_kaspi_fields
```

If running in CI/deployment context, use:
```bash
npx prisma migrate deploy
```

After migration, regenerate the client:
```bash
npx prisma generate
```

### Task 7 — Fix TypeScript compilation errors
After schema change, TypeScript will report errors in files that reference the removed fields:
- `app/api/bookings/route.ts` — references `kaspiMerchantId`, `kaspiApiKey`, `paymentInvoiceId`, `paymentExpiresAt` + imports `createKaspiInvoice`
- `lib/actions/payment-settings.ts` — likely references `kaspiMerchantId`/`kaspiApiKey`
- `app/dashboard/settings/billing/billing-content.tsx` — references `kaspiMerchantId`, `kaspiApiKey`
- `components/service-form.tsx` — may reference Tenant-level `requireDeposit`
- `__tests__/payment-surface.test.ts` — Kaspi payment test assertions

For this plan, add `// TODO(12-02)` comments at error sites rather than full fixes — the full fixes happen in Plans 12-02 and 12-04 respectively. **Exception:** if the TypeScript error is a simple field removal (field no longer exists on the type), remove the field reference.

Run `npx tsc --noEmit` to surface all errors. Triage:
- Simple: field removed from type → delete the field reference
- Complex: full function removal → leave `// TODO(12-02)` or `// TODO(12-04)` comment

### Task 8 — Update test file: remove Kaspi invoice assertions
In `__tests__/payment-surface.test.ts`, find and remove or update tests that assert Kaspi-specific behavior (e.g., `invoiceCreated: true`, `paymentInvoiceId`, `createKaspiInvoice` mock). These tests will fail after the schema migration. Replace with a `// Removed in Phase 12: Kaspi deposit flow` comment or delete the test block entirely if it has no non-Kaspi assertions.
